# Project: 3-Tier Notes Application (Kubernetes Orchestration Demo)

--------------------------------------------------
1. APPLICATION OVERVIEW
--------------------------------------------------
This project is a 3-tier web application deployed on Kubernetes to demonstrate:
- Container orchestration
- Configuration management
- Secrets handling
- Service-to-service communication
- Persistent storage usage

Tiers:
1. Frontend (Nginx + HTML/JS)
2. Backend (Node.js + Express)
3. Database (MongoDB)

--------------------------------------------------
2. COMPONENTS & RUNTIME PORTS
--------------------------------------------------
Component   | Technology            | Container Port | Exposed Via
------------|------------------------|----------------|-------------
Frontend    | Nginx                  | 80             | Ingress / NodePort
Backend     | Node.js (Express)      | 5000           | ClusterIP
Database    | MongoDB                | 27017          | ClusterIP

5. PERSISTENT STORAGE (DATABASE â€“ STATEFULSET)
--------------------------------------------------
MongoDB is deployed using a Kubernetes StatefulSet to ensure:
- Stable network identity
- Stable pod naming
- Dedicated persistent storage per pod

Resources Used:
- StatefulSet            : Manages MongoDB pod lifecycle
- VolumeClaimTemplates   : Automatically creates PVCs
- PersistentVolume (PV)  : Underlying storage provisioned via StorageClass
- StorageClass           : Handles dynamic volume provisioning

Storage Behavior:
- Each MongoDB pod gets its own PVC
- PVCs are not deleted when pods restart
- Data remains intact across pod restarts and rescheduling

Example:
- Pod Name : mongo-0
- PVC Name : mongo-data-mongo-0

MongoDB data survives:
- Pod restarts
- Node rescheduling
- Application redeployments

This setup reflects production-style stateful workload management.
